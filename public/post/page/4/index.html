2021-7-21

为了测试python在web服务器上运行，装了个httpd。

过程又不怎么顺利，出了些比较烦人的问题，主要有三个：

  • 一个是配置文件的权限问题。

    配置文件httpd.conf有两个，一个是出现在用户目录下的安装目录下，一个是出现在/etc/httpd/conf/。每次都喜欢去修改用户目录下的那个，但实际上起作用的却是etc目录下的那个，最后只好把etc下面的那个改名做了个备份，再把用户目录下的用ln -s命令链接到etc目录下。这以后不管修改哪一个，两个都一起变动。

  • 另一个是cgi的模块载入问题。

    用hello.py做测试，浏览器输入http://127.0.0.1/cgi-bin/hello.py后，显示的是源代码，不是编译后的内容。改了好久都不行，最后发现问题出在cgi模块未载入。这得在httpd.conf配置文件里改，找到LoadModule cgid_module modules/mod_cgid.so那一行，把前面用来注释的#号去掉。

  • 再一个就是编译出错导致的一个bug，并不一定每次编译都出现，但只要出现就必须用下面的方法去解决。

    问题是必须把httpd.conf放在系统根目录下一次，才有可能用系统服务的方式去启动那第一次。启动后又可以再删除那根目录下的配置文件，接下来这问题便没了，自然恢复到正常情况。问题不再是问题。

重装i2p又踩了很多坑，而且和上一次有很大不同。好处是以i2p认识更深了。

主要有几个：

  • i2p有几个版本，两个java版，一个c++版。C版的是安装i2pd，速度最快，占用系统资源最少，但用户界面不友好，一般用户很难去控制。因为交互不简单，不友好，我只能忍痛放弃，换回讨厌的java版。
  • java版有两个安装包，一个是i2p，一个是i2p-bin。安装后从任务管理器来看，前者占用系统资源似乎相对少那么一点点。所以，i2p成了我最终的选择，试了又试后放弃i2p-bin。
  • 安装后，类为用户权限的限制，i2p的运行在建立pid文件时会遇上一个错误。这算是一个老问题，很多linux软件都会遇上这问题。我最后是把i2p的目录用chown yoli:root i2p -R，改了下。再改了下i2p目录下的i2prouter文件中的内容，把I2P_USER="i2p"改为I2P_USER="yoli"。再改了下/usr/lib/systemd/system目录下的服务文件i2prouter.service的内容，把User=i2p改为User=yoli，又增加了一个Group=root。再接着，便是把/run/下的i2p目录用chown yoli:root i2p -R，改了下。这样后基本上能正常运行了，但重启后就又不能正常运行。因为run目录是运行在内存中，是临时目录，一重启里面的内容便会被自动清除掉。
  • 试着把pid文件放到别的目录中去，更改配置文件和服务文件。结果事实证明不管用，因为即使改了配置文件，还是会有要使用到run目录下的pid文件。怀疑是jar在调用，如果真是它，那除非把jar全改了，否则还是会继续bug下去。这条路显然太麻烦，就算能行得通也不实际，不如不用。
  • 在/etc/tmpfiles.d下，我新建个conf文件，名叫i2p.conf。里面的内容就一句：d /run/i2p 0775 yoli root。有了这个，重启后系统就会自动在run目录下按我的要求新建一个i2p目录，如此便初步解决了pid存放的难题。
  • 回过头来一想，group再设置为root还有那意义吗？不如都改回yoli得了。

上面的这些坑都还不算什么，i2p最大的坑和amule最大的坑一样是kad的防火墙问题。因为宽带网络运营商不提供公网IP，再加上运营商防火墙跟政府防火墙的阻拦，kad网络基本被封个半死不活，i2p同样因此而受害，用起来极为不畅。

2022-9-26

时隔一年，今天又重装了一次i2p，对它的认识更深了。

其实，i2p和yggdrasil说白了都是让用户在访问因特网时先访问它们，再通过它们去访问因特网。它们不像一般的代理，是建立在因特网上。i2p的代理是建立在i2p网上，yggdrasil代理是建立在yggdrasil网上。它们既可以独立于因特网外，又可与因特网连接在一起。

于是，通过它们去上网，既能上i2p或yggdrasil网，也可以上因特网。

但那不是无条件的，我们必须安装i2p或yggdrasil的软件。这是基础条件，有了它我们才能上i2p或yggrasil网。

这类网最大的特点是分布式，不再像因特网那样把数据集中在中心服务器。在分布式网络上，我们每一个网民都是服务器，也是客户端。我们可以在自己的电脑上建网站，不需要去别人那里申请。我们可以在自己电脑上建代理，然后给别人用。而这对于那些需要翻墙的人。确实是一个神器。

可惜的是，在速度上i2p代理远不如yggdrasil代理。但i2p和tor跟yggdrasil都能用来翻墙。

如何用i2p翻墙，一般用户可能会看不懂。虽然在看得懂的人看来很简单，无非是两步，一步是安装i2p，第二步是设置i2p代理。事实上对初学者，对不懂的人而言，个人感受是非常不友好。

我这次遇上了个bug，花了不少时间才解决。这个bug不是由i2p代理本身引起的，而是firefox浏览器的代理扩展引起的。后来我试着不用那代理扩展，而是直接在浏览器的设置里设置代理，结果能过了。这么一来自然就知道问题出在哪里了，接着把那代理扩展的设置里有关i2p的设置给删除，再重新设置一次。虽然设置的内容是一样的，但这次却轻易地通过了。之前是怎么都通不过，这就是BUG，错得很无理。

在ZeroNet网络上，自己的用户和网站的注册，凭据是自己的私钥。这一点至关重要!!!

私钥丢了就再找不回来了，自己在零网上的帐号就等于不再是自己能动的了，自己的网站也不再是自己能动的了。我就是手快忘了这一点，把原来的私钥文件给删除了，帐号yoli和原来用这号建的那些网站就这么全丢了。

备份你的ZeroNet帐号/私钥：ZeroNet\data\users.json这个文件相当于你的比特币钱包，这个文件丢失/损坏而你又没备份的话，你就无法操作你的帐号/网站了。所以把这个文件备份出来很重要。恢复/重新安装/迁移/跨设备使用ZeroNet只需把这个文件放到原位覆盖，方法是退出ZeroNet后台程序（托盘区0图标上点右键，选择Quit）再覆盖，覆盖完以后再启动ZeroNet。注意：一定要等到各网站数据全部同步完成再发帖，否则你会对空数据/旧数据签名发布到全网，导致你的数据丢失！**

可以说，私钥是一把双刃剑，越锋利就越危险。

好在零网上的网站可以克隆，注册ID的网站自然也在克隆的范围内，原来丢掉的网站自然也在克隆的范围内。所以说，也不是完全找不回来了，只是麻烦点。这也是去中心化网络与原来网络的不同。

但要让克隆的网站像之前的网站一样好用，就不是那么简单的事情了，中间应该还有很多细节的工作要处理。总之不像宣传的那么简单，好像一键处理就行。

或者说克隆的只是个空壳，用户、权限、数据等统统不管，否则就应该啊迁移。换言之，迁移才是完整地克隆。

2023.6.12 注册ID

ZeroNet是怎么做到去中心化的帐号验证的？注册帐号有什么注意事项？

你第一次运行ZeroNet时本地生成一个主私钥，这个主私钥存在本地/ZeroNet/data/users.json里。注册时访问ZeroID.bit，就会由主私钥生成那个网站的auth_address和auth_privatekey作为你在那个网站的唯一身份凭证，同样存在本地/ZeroNet/data/users.json里（auth_address和auth_privatekey访问网站后由主私钥自动重新生成，不用担心丢失）。目前需要有个服务器，向你的auth_address颁发证书并写入到P2P网络，之后你使用证书验证就不关那个服务器的事了，不需要跟它通信，也就是说只在第一次注册时才会跟那个服务器有通信。目前注册时向那个服务器通信有两种方式，一种是浏览器发送http请求，另一种是比特信。如果有匿名需求的话，注册时要用Tor。

我的ZeroID会不会被删除？删除后有什么影响？

明网网站的所有者就是上帝，想删除你的帐号就删除你的帐号。但零网不一样，一旦你成功注册了帐号，这个帐号就永远是你的了，没有任何人能删除你的帐号，包括给你提供帐号注册服务的网站。这意味着你可以在任何支持该ID的网站中发帖发视频发评论。当然你发帖所在的网站所有者，仍然有权利禁止你在他的网站发布内容，但他没权利也没能力删除你的帐号，因为你仍然可以使用这个帐号在别的支持该ID的网站中发贴。

为什么ZeroNet注册ID不使用区块链技术？

blockchain注册有一定缺点：

  • 新用户需要等待区块链下载完成后才能注册

  • 需要块被确认后才能使用ID，大约要等几分钟，有时候更长

  • 很难防spam

这东西怎么连个密码都没有？

私钥比密码安全。

ZeroNet是什么？简单介绍

零网本质上就是一个BitTorrent网络 + 本地网站服务器。什么意思呢？就是通过零网，A在本地电脑上运行一个网站服务器AS，A不仅可以通过浏览器AC访问A自己建的网站AW，还可以将网站内容文件AW通过BT网络分发到其他人电脑里，B通过B电脑上的网站服务器BS解释A传来的网页文件AW，然后B通过浏览器BC访问网站服务器BS来查看网站AW。当A修改网站内容AW时，网站的变动AW2将通过BT网络传到B电脑里，这样B就能看到A更新后的网站AW2了。当然A也可以对创建的网站AW设置权限，允许别人将他们创建的文件增加到A的网站文件AW中（比如，博客评论），于是B对A网站添加的网站文件AWb将通过BT网络分发且被其他人C认可；当A电脑收到B新添的文件AWb后，A电脑上的网站服务器AS会一起解释原来的网站文件AW和新添加文件AWb，这样A通过浏览器AC访问网站服务器AS时，就会发现B在A博客文章（本质就是原来的网站文件AW）下发表了评论或表情图片等（本质上就是B对A网站添加的文件AWb）。

用tor翻墻得经历从慢到快的过程，快了后要持续快就不能再随便改动网桥的设置。找到了好的网桥，快起来后就轻易不要改动。改动后就又得慢上一阵子了，不知什么时候才快得起来，一切又得从头来始，从慢到快慢慢来……

最慢是第一步突破封锁，一旦改动了网桥的设置，就又得从头开始突破。

2021-7-30

tor browser

今天发现不只是改动网桥设置的问题，有时候改动设置反而能加快连接，这得看具体是什么情况。

  • 如果只是改动添加网桥里面的内容，而不改用不同的网桥方式，反而有可能使本来卡着的进度条快速突破。这也突破封锁最难的一步。具体的操作，其实也不需要真正改动里面的内容，只需把排在后面的某条改动到第一条，这样就行了。
  • 如果一定要改用不同的网桥方式，最好先把配置文件备份torrc。这文件先不要直接改动，直接改动会跟浏览器的自动修改互抢，抢来抢去里面的内容就容易出问题。如果一定要手动改动，也不是没办法，只要记得先把浏览器关闭就行。这样就不会再互相抢着修改里面的内容了。

2021-9-21

蓝灯

今天终于在linux上把蓝灯给搞定了，实现了浏览器之外的翻墙。

蓝灯装上后，最开始的速度还真是快得吓人，可惜高速免费听说只有500M/月。之后便会进入到正常速度的翻墙。

蓝灯的安装文件是deb文件，得先用debtap这命令转化，之后才能用pacman -U命令来安装。

安装后，双击它发现还不能正常运行。于是进入终端用命令运行来查看，发现提示说是有少了一个.so的库文件。

到/usr/lib下搜索同名文件，发现了一个只版本不同的高版本文件。用ln -s命令给那文件创建一个新的软链接，链接的文件名自然是那个少了的低版本文件名。接着一测试，蓝灯能正常运行了。

——上面用软链接改文件名的方法是在外文网站上学到的，不然还真没想到。

如果命令行工具也想访问蓝灯网络，需要为 Shell 设置两个环境变量 HTTP_PROXY 和 HTTPS_PROXY。

找到蓝灯在本地启动的具体端口号。打开蓝灯，依次选择 Settings -> ADVANCED SETTINGS 即可看到蓝灯在本地选择的端口号。

然后去 Shell 里执行以下两个命令设置环境变量：

export HTTP_PROXY=http://127.0.0.1:43385
export HTTPS_PROXY=http://127.0.0.1:43385

就完成配置了。

最后为了避免每次都要敲这么长的命令，写一个 Shell 函数：

# http proxy util
hp() {
  if [ "$1" = "enable" ]
  then
    PORT="43385"
    if [ -n "$2" ]
    then
      PORT="$2"
    fi

    export HTTP_PROXY=http://127.0.0.1:$PORT
    export HTTPS_PROXY=http://127.0.0.1:$PORT
  else
    export HTTP_PROXY=""
    export HTTPS_PROXY=""
  fi
}

把这都函数代码放到 Shell 启动脚本里。然后需要开启代理的时候执行 hp enable 即可，或是指定代理端口号 hp enable 33333。关闭代理执行 hp 即可。
